# IFCXS 記法 v0 (draft)

状態: 素振り。この文書は二室一扉を書くために生まれた最初の記法の記録であり、確定仕様ではない。ここでの決定の理由は docs/decisions/0001-notation-v0.md に、思想は docs/writing-architecture.md にある。

設計基準はひとつに固定する。**LLMが一つのコンテキストで読み切り、正しく部分編集できるか。** この基準から、テキストネイティブであること、参照が人間の読める階層パスであること、小さなファイルであること、単位と座標系を先頭で一度だけ宣言することが導かれる。

## モデル

一次要素は空間である。`space` は室・ゾーン・外部の領域を表し、階層パスで名指される。壁はそれ自体が独立した物ではなく、二つの空間の境界という関係 (`boundary`) として持つ。境界には厚み・仕様・耐火などの性能が属性として乗る。開口 (`door` / `window`) は境界に開いた接続である。骨格はグリッド (`grid`) とレベル (`level`) が与える。型は巨大なクラス階層ではなく開かれた語彙であり、`room` `corridor` `exterior` などのコア語彙に自由な語を足してよい。

形はこのソースには無い。壁の線分は空間の割付から導出され、平面図は必要になった時に生成される。同じ構成から複数の形が出ることは欠陥ではない。

## 記法 (author形式: 独自DSL)

```
# 二室一扉 — 最初の一手

ifcxs 0.1
name 二室一扉
unit mm

grid X 0 3600 7200          # 通り芯 X1, X2, X3
grid Y 0 4500               # 通り芯 Y1, Y2
level L1 0 h:2400           # FL高さ 0、基準天井高 2400

space /L1/a room X1..X2 Y1..Y2 name:居室A
space /L1/b room X2..X3 Y1..Y2 name:居室B
space /out  exterior name:外部

boundary /L1/a /L1/b t:120 spec:PW1
  door w:780 h:2000

boundary /L1/a /out t:150 spec:EW1 fire:60
boundary /L1/b /out t:150 spec:EW1 fire:60
  door w:900 h:2100 edge:S name:玄関
```

文法は次の通り。一行が一文であり、`キーワード 位置引数... key:value...` の形をとる。`#` 以降はコメント、値に空白を含めるときは `"..."` で囲む。字下げした行は直前の `boundary` に属する開口 (`door` / `window`) である。

`grid X 0 3600 7200` は昇順の座標 (mm) を宣言し、通り名 X1, X2, ... が自動で振られる。`level 名 z [h:天井高]` がレベルを宣言する。`space パス 型 [X?..X? Y?..Y?] [属性...]` は空間を宣言し、パスの先頭セグメントがレベル名ならそのレベルに属する。領域は v0 では通り芯で囲む矩形ひとつに限る。`exterior` のように領域を持たない空間も書ける。

`boundary パスA パスB [t:厚 type:wall|open 属性...]` は境界を宣言する。境界の壁芯線分は書かない — 二空間の割付から導出される。接していない空間の境界はエラーになる。`type:open` は垂れ壁のない開放境界で、扉なしで通行できる。外部のような領域なし空間との境界は、部屋の外周から他室と接する区間を除いた残りとなり、複数辺に分かれるため、開口を置くときは `edge:N/E/S/W` で辺を指定する。開口の位置は `at:0..1` (既定 0.5) で線分上に置く。

既定値: 境界は `type:wall`、開口位置は `at:0.5`、描画時の壁厚は t 未指定なら 100mm。面積は壁芯で算定する。

## 書き比べ — 同じ二室一扉を三つの形式で

v0 を決めるために、同じ場面を三つの形式で書いた。上の DSL が一つ目である。

二つ目、YAML:

```yaml
ifcxs: "0.1"
name: 二室一扉
unit: mm
grid:
  X: [0, 3600, 7200]
  Y: [0, 4500]
levels:
  L1: {z: 0, h: 2400}
spaces:
  /L1/a: {type: room, at: [X1, Y1, X2, Y2], name: 居室A}
  /L1/b: {type: room, at: [X2, Y1, X3, Y2], name: 居室B}
  /out:  {type: exterior, name: 外部}
boundaries:
  - between: [/L1/a, /L1/b]
    t: 120
    spec: PW1
    openings:
      - {kind: door, w: 780, h: 2000}
  - between: [/L1/a, /out]
    t: 150
    spec: EW1
    fire: 60
  - between: [/L1/b, /out]
    t: 150
    spec: EW1
    fire: 60
    openings:
      - {kind: door, w: 900, h: 2100, edge: S, name: 玄関}
```

三つ目、JSON。これは `ifcxs json` が吐く正準形そのものであり、全文は examples/two-rooms.canonical.json にある (111行)。抜粋:

```json
{
  "ifcxs": "0.1",
  "name": "二室一扉",
  "unit": "mm",
  "grid": { "X": [0, 3600, 7200], "Y": [0, 4500] },
  "spaces": {
    "/L1/a": { "type": "room", "at": ["X1", "Y1", "X2", "Y2"], "attrs": { "name": "居室A" } }
  },
  "boundaries": [
    { "between": ["/L1/a", "/L1/b"], "kind": "wall", "t": 120,
      "openings": [{ "kind": "door", "w": 780, "h": 2000, "at": 0.5 }] }
  ]
}
```

| 観点 | DSL | YAML | JSON |
|---|---|---|---|
| 同じ場面の行数 | 14 (コメント除く) | 約30 | 111 |
| 人間の読み書き | 図面の密度に最も近い | 十分読める | 構造は明快だが冗長 |
| LLMの部分編集 | 一行一文で差し替えが局所的 | 良好 | 良好だが括弧の整合が絡む |
| diffの言葉 | 「この室が広がった」が一行で出る | 良好 | ノイズ多め |
| パーサ | 自作 (約300行で済んだ) | 既存 | 既存 |
| レイヤー合成 | 直接は不向き | 中間 | 最も素直 (IFCX/USDと同じ土俵) |

**決定: author形式はDSL、機械形式は正準JSON。** 人が(そしてLLMが)読み書きする原本はDSLで持ち、合成・diff・外部接続の土台には `ifcxs json` の正準形を使う。YAMLは両者の中間で決め手を欠くため見送る。DSLの代償はエディタ支援が無いことと文法の保守だが、文法を「一行一文 + key:value + 字下げ」から増やさないことで抑える。

## 正準JSONの形 (機械形式)

`ifcxs json` が吐く正準JSONは次のキーを持つ。`ifcxs` (版)、`name`、`unit`、`grid` (X/Y各軸の座標配列 mm)、`levels` (名前→ `{z, h?}`)、`spaces` (パス→ `{type, at?, attrs?}`。`at` は `[XA, YA, XB, YB]` の通り名4つ)、`boundaries` (配列。各要素は `{between, kind, t?, edge?, attrs?, openings?}` で、`between` は昇順に並べた2パス、`openings` の各要素は `{kind, w, h?, at, edge?, attrs?}`)。

安定性の規則: すべてのオブジェクトキーはソートされ、`spaces` はパス順、`boundaries` は `between` の辞書順に並ぶ。同じ構成からは常にバイト同一のJSONが出るため、diff・ハッシュ・レイヤー合成の土台に使える。素振り段階の現在は src/model.ts の `toCanonical()` の実装を正とし、本節はその写しである (乖離したらどちらを直すかをその時に決め、ADRに残す)。

## v0.1 追補 — オフセットと高さ方向 (2026-07-23)

複数レベル・廊下と複数室・芯外の壁を書くための追補。理由は ADR-0002、実例は examples/office.ifcxs (2フロアのオフィス、約100行)。

**通り芯オフセット。** 領域の参照は `X2+600` `Y3-150` のように芯からの寸法を足し引きできる。座標の直書きは無い — 芯の言語を保つ。

**レベルと高さ。** `level L2 4000 h:2700 slab:1300` — z (FL)、h (基準天井高)、slab (その階の床組み厚: 下階天井面から自階FLまで)。空間は `h:` 属性で天井高を個別に持てる。checkが不変量「天井高 + 上階slab ≤ 階高」を検査し、超えると食い込みエラー。`levels` コマンドがテキストの矩計として積み上がりを表示する。最上部に空間を持たないレベル (屋根面) を宣言すると最上階にも上限が効く。

**垂直の境界。** 床は書かない — 垂直の隣接は平面の重なりから導出され、既定は床である。例外だけを書く: `boundary /L1/stair /L2/stair type:stair` (扉0枚で通行可)、`type:shaft` (EV等 — 連続するが通行不可)。壁境界 (`wall`/`open`) を異なるレベル間に書くとエラー。吹抜けは `type:void` として設計予定 (未実装)。

**基本計画の解像度。** このデータが表現するのは計画初期の解像度であり、垂れ壁・建具詳細・仕上げは持たない。`type:open` の境界は「垂れ壁があるか」を言わない — その決定は後段に属する。平面図では開放的な分節を破線で描く。

**stats の use 集計。** 空間に `use:rentable` / `use:common` などの属性を与えると、statsが面積比 (レンタブル比) を返す。

**数えない分節 (area / seg)。** 室を割らずに床材の切替や壁材の途中変更を書く従属要素 (ADR-0003)。`space` の字下げに `area X?..X? Y?..Y? floor:モルタル name:土間`、`boundary` の字下げに `seg at:0.75 w:3600 spec:ガラスパーティション` (位置は開口と同じ流儀)。隔離則: これらは面積・室数・グラフに一切影響しない。リトマス試験は「面積表に一行として現れてほしいか」(欲しければspace) と「そこを通れるかが問題になるか」(なるならboundary/opening)。平面図ではareaは淡い面+破線、segは壁の色調変化として描かれる。

## この数十行で決まったこと

原稿の予告どおり、二室一扉を書くだけで四つの設計判断が言葉になった。空間の同一性は人間可読の階層パスに与える。境界はどちらの空間にも属させず、二つのパスを結ぶ第一級の関係とする。接続は境界上の開口とし、垂れ壁のない分節は `type:open` で表す。パスは `/レベル/空間` で切り、レベルに属さない外部はサイト直下に置く。それぞれの理由と代替案は ADR-0001 に記録した。

## 開いている問い

吹抜けと階をまたぐ空間、半屋外と外部空間の分節 (`/out` が一枚岩で良いはずがない)、室とゾーンの二層粒度 — これらは一フロアを書くとき (M1) に正面からぶつかる。記法側では、矩形ひとつに限った領域 (L字型の室が書けない)、開き勝手の記述 (v0は「先に書いた空間側へ開く」という描画規約に逃げている)、建具語彙の粒度、壁厚の振り分け (v0は芯振り分け固定)、面積規約 (v0は壁芯固定) が残っている。IFCのGUIDとの対応は、取り込み (M5) の際に `id:` 属性のレイヤーとして後置する。
