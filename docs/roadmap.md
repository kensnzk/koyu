# koyu ロードマップ

原稿 (writing-architecture.md) の「最初の一手と、その先」を行程に切ったもの。Linear の koyu プロジェクトのマイルストーンと対応する: https://linear.app/munipersonal/project/koyu-2789f588a03a/overview

これは探求であり、締切は設けない。各段の終わりにあるのは日付ではなく分水嶺の問いで、答えられたら次へ進む。

## M0 素振り — 二室一扉 (2026-07-22 ひと通り)

IFCXのサンプルを読み (ifcx-notes.md)、二つの室とその間の扉一つを自分の記法で書いた。空間の同一性・境界の帰属・接続・パスの切り方はここで決まり (decisions/0001)、記法の書き比べと決定は spec/notation-v0.md に残した。最小ツール (パーサ・整合チェック・グラフ問合せ・平面SVG生成・正準JSON) が動き、テスト14本とCI設定がある。分水嶺の問い「数十行で四つの決定が言葉になるか」には、なった、と答えられる。残: IFC5 alphaビューアでのレイヤー合成の実見。

## M1 一フロア〜複数フロア — 曖昧さへの正対 (2026-07-23 前半済)

一住戸〜一フロア以上を書き、IfcSpaceが実務で機能しにくかった当の曖昧さに定義を与える段。位置づけをここで明確にした: **このデータの解像度は基本計画レベルである。** 垂れ壁や建具詳細を表現しないのは省略ではなく抽象度の選択であり、計画初期にBIMが重すぎるという既存の弱点の裏側こそがこの記述の主戦場になる。使い勝手の参照点はVectorworks的な軽さ。

前半 (済): 2フロアのオフィス (廊下+複数室+コア) を約100行で記述。分節を「数える分節」(室・境界) と「数えない分節」(`area`=室内の床材切替等、`seg`=壁材の途中変更) に二分し、後者は面積・室数・グラフに影響しない隔離則を敷いた (ADR-0003)。通り芯オフセット (`X1+3200`)、複数レベル、高さ方向の一貫性 — BIMが3Dであることで暗黙に担保していたもの — を「宣言された不変量 (天井高+上階slab ≤ 階高) と自動チェック」に置き換えた (ADR-0002)。床は書かず垂直隣接は導出、例外 (`type:stair` / `type:shaft`) だけ宣言。`levels` コマンドがテキストの矩計。避難の問い「2階執務室から外へ扉4枚 (階段経由、EVは通れない)」に即答できる。

後半 (進行中): 10階建て内廊下型集合住宅43戸 (1F共用部・2〜9F基準階・10F PH・屋上、EV1基+屋外階段) を159行で記述 (2026-07-23)。基準階の反復は「一度だけ書く」機構 — レベルの等差宣言・パスのスパン展開 (/L2..L9/A)・垂直積層のstack宣言 (ADR-0004) — が吸収し、159行が90空間・244境界・10フロアに展開される。屋外階段は半屋外の初出 (上階は手すり壁で外気に接し、地上階はopenで外部へ)。「9階の住戸から地上まで扉2枚 (玄関+階段防火戸)」に即答できる。

続き (2026-07-23 後半): 間取り・L字・吹抜け・採光を一気に通した。Aタイプ住戸に間取り (L字LDK・洋室・水回り・玄関廊下) を書き、`zone` (数える集約 — 幾何なし・面積は配下の合計・use継承) で住戸の言葉を保った (ADR-0005)。吹抜けは `type:void` の垂直境界 = 高さ不変量の宣言的免除で、オフィスのエントランスを2層吹抜けに (ADR-0006)。例外階はスパン分割で新記法なしに書ける。`light` コマンドが住居系居室の1/7採光を粗く判定し、54㎡住戸の開口不足を実際に指摘した — 最初の法規クエリ。集合住宅は183行→114空間・316境界、テスト56本。

続き2 (2026-07-23 深夜): 半屋外・語彙の台帳・開き勝手・メゾネット・部分吹抜けを通した。半屋外は宣言ではなく導出 — 外部に対しopenまたはair:1 (手すり等、遮蔽しない物) の境界を持つ空間 (ADR-0007。kindにrailを足す案はIFC参照で棄却: IfcRailingは要素であり、境界関係はPhysical/Virtualしか言わない)。属性の付け方は語彙の台帳 (spec/vocabulary.md) を契約に (ADR-0008)。開き勝手はhinge/swing。メゾネットは「パスの第一義は集計の階層」— level:属性で階を明示し、ゾーンが階を跨ぐ (examples/house.muro)。部分吹抜けは被覆率で不変量免除を制御 (ADR-0006追記)。Aタイプにバルコニー追加、採光はバルコニー越し0.7掛け。テスト63本。

続き3 (2026-07-23): 明示位置・建具アセット・合成 (ADR-0010)。開口の位置が通り芯の言葉で書けるようになり (`at:Y2+1820`)、はみ出し・軸違い・開口同士の重なりが検査される。建具の型は `asset` (RevitのFamily、USDのReference — インスタンスが上書き) と `style:hinged/sliding/auto` (平面の建具表現が変わる)。そして `import` — base層が基盤 (grid/level) を一度だけ宣言し、アセット・敷地・各フロアを別ファイルで分担して加算合成する。衝突 (パス・アセット名・grid再宣言) は出所つきのビルドエラーで、`koyu check main.muro` が一棟のビルドの門番になる。examples/house/ (5ファイル) が実証 — 単一ファイル版と同じ答え。テスト79本。

続き4 (2026-07-23): 敷地形状とショーケース。敷地はまず矩形にならない — 敷地の形は設計の生成物ではなく測量由来の所与なので、唯一「書かれる形」として `polygon` を認め、隔離レイヤー (site-geometry.muro) に置いてimportする (ADR-0011)。導出面積はシューレースで宣言 (測量値) と照合、建物のはみ出しはエラー、配置図に一点二点鎖線の敷地境界線。そして**全機能のショーケース examples/tower/** — 低層商業+高層住宅の11階建て複合ビル (延床4,785.92㎡・敷地は五角形1,097.80㎡の角地)。8ファイル合成、2層吹抜け、噛み合うL字住戸、バルコニー、低層部屋根のテラス (例外階=差分レイヤー)、PH、アセット、明示位置、Aタイプ間取り。178空間・542境界が警告ゼロ、居室66室の採光、「9階LDKから道路まで扉4枚」。テスト98本。**今後の検討の土台はこの建物。**

残: 外部の分節の続き (延焼ライン・斜線への足がかり)、屋根の有無 (半屋外の細分)、建具語彙の残り (折戸・防火設備の別)。分水嶺: これらを書き切っても記法が破綻しないか — 現時点で破綻の兆候なし。

## M1.5 直交からの最初の逸脱 — 斜め (判断: 見送り)

2026-07-23 判断: 雁行・セットバック・コアの振れといった「斜めに見える大半」は矩形の合併とオフセットで書ける。真の非直交は、共有辺・外周・重なりを支える矩形の区間代数を多角形クリッピングに全面書き換えることを意味し、「簡単なら」の範囲を超える。曲線と同じ「扱わないこと」側に寄せ、実案件の平面が矩形合併で書けなくなった時に多角形領域として一括で設計する。

## M2 一棟 — グラフへの問い

一棟を書き、グラフに問いを投げる。この室からこの室まで扉をいくつ通るか、防火区画をまたがずに外へ出られるか、共用と専有の面積比はいくつか。**部材のモデルなら抽出作業が要る問いに、何もせず答えられるかどうかが、オントロジーとしての分水嶺** (原稿)。避難・面積・区画・動線が同じ記述の異なる読み方になることを確かめる。

## M3 平面図 — 実務の言語への翻訳

素描 (M0のSVG) を図面に近づける。寸法線、通り芯記号、建具の作図表現、縮尺。平面図が生成できれば人に見せられ、実務の言語に翻訳できる。生成が一意でないこと (同じ構成から複数の形が出ること) をここで実際に確かめる。

## M4 版管理とレビュー

テキストであることの帰結を回収する。差分を「この室が広がった」という構成の言葉で表示するdiffツール、Pull Requestによる設計レビューの試行、コミットごとに整合と法則を確かめるCI (芽は .github/workflows/ci.yml と `koyu check` にある)。分担の器は先にできた — import合成 (ADR-0010) でフロア・敷地・建具が別ファイルに割れるので、レビューの単位とファイルと分担の単位が一致する。

## M5 IFC取り込み — コーパスの獲得

一方向・非可逆のIFC→koyu変換を書く。往復互換は最初から捨てる。BIM/IFC系のツールはThatOpenのOSS (web-ifc, @thatopen/components) を使う。コーパスの最初の候補は手元の IFC_samples/ にあるBLCJサンプル (RC造3000㎡・S造1000㎡)。IfcSpaceが書き出されていないモデルから空間をどう復元するかが、おそらく本当の難所。参照リポジトリをクローンする場合は ~/Documents/github に置く。

## 横断: IFCカバレッジの運用 (2026-07-23〜)

IFC4建築系コアを鏡にして、書けるもの・まだ書けないもの・方針として書かないものを [docs/ifc-coverage.md](ifc-coverage.md) で管理する。未対応 (○)・部分対応 (◐) はLinearの別プロジェクト [koyu × IFC カバレッジ](https://linear.app/munipersonal/project/koyu-ifc-カバレッジ-c97cdb2fdc18) のIssue (概念クラスタ単位、15件) と対応。実装や決定が動いたら表を先に直し、Issueを閉じる。当面の優先はP2の4件: 敷地とその境界 (MUN-149 ✔ 2026-07-23済)、外部空間の分節 (MUN-150 ✔ 同)、外構 (MUN-151)、立ち上がり・パラペット (MUN-152)。149/150はADR-0009 — /outの分節 (road:幅員)、地上外部空間のタイル、zone site:1の宣言/導出照合、`site`コマンド (敷地面積・接道・建蔽率・容積率)。戸建例のL1平面は配置図兼用になった。

## 地平 (マイルストーンにしない遠景)

デジタルツイン (実測を設計の上に重ねる一枚のレイヤー)、都市 (数百行の軽さでCityGML・歩行空間ネットワーク・不動産IDと同じ層に載る)、フィジカルAI (空間グラフ+通行可能性+意味がそのまま行動の地図になる)。いずれも原稿の「地平」の節が原典。M2のグラフが答えられるようになったとき、最初の接続実験を選ぶ。

## 扱わないこと (再掲)

オーサリングツールは作らない。往復互換は捨てる。曲面と自由形状は対象外。設計意図の記録はデータではなくプロンプトの側。事業化は当面目的にしない。
