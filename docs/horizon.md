# 地平の設計 — デジタルツイン・オントロジー・LLM接続

日付: 2026-07-23 / 状態: 構想 (ADR-0012がLLM接続の第一手を採用)

原稿「建築を書く」の「地平」の節を、v0.7の実装 (合成・出所・正準JSON・敷地形状・ショーケース) の上に具体化する。三軸に共通する原則を先に言う: **koyuは小さいままでいる。** 接続はkoyuの中に機能を増やすことではなく、koyuが既に持つもの — 人間可読パス・空間グラフ・正準JSON・checkの門番 — を外の世界の結節点にすることで実現する。

## 軸1: LLM接続 — 建物をエージェントの作業対象にする

設計基準「LLMが一つのコンテキストで読み切り、正しく部分編集できるか」は数字で答えが出た (examples/comparison/README.md): 延床4,786㎡の一棟が原本8,099トークン、同じ場面でIFC4は14倍・IFCXは25倍、実務のIFC出力はコンテキストに載らない。ここから先は「読める」を「働ける」にする段である。

**第一手: MCPサーバー (ADR-0012 — 実装済み)。** `koyu-mcp` がstdioでモデルを公開する。ツールは要約 (`model_summary`)・検査 (`check`)・レイヤーの読み書き (`layers` / `write_layer`)・グラフの問い (`doors` / `stats` / `light` / `site` / `levels`)・平面生成 (`plan_svg`)。エージェントの作業ループはgitのそれと同型になる: **読む → レイヤーを書き換える → checkが門番として通す/弾く → 問いで帰結を確かめる。** 書き込みは.muroに限定し、履歴はgitが持つ (OwnerHistoryを持たない方針の帰結がここで効く)。依存ゼロの方針を守るため、MCPはSDKなしの手書きJSON-RPC 2.0で実装した — パーサを自作した時と同じ賭けである。

**第二手: エージェント評価 (eval)。** 「Bタイプを2住戸に割れ。採光と避難を保て」という課題を与え、check/light/doorsをオラクルに合否を測る。記法の設計 (一行一文・出所つきエラー・宣言と検査) がエージェントの成功率にどう効くかを測定できる形にする — 論文の実験の柱。towerがベンチマーク盤面になる。

**第三手: 差分の言葉 (M4と合流)。** `koyu diff` が「/L5/B が北に600広がり、Cが縮んだ。専有面積は±0」を出せば、エージェントの自己検証とPRレビューが同じ言葉を使える。正準JSONの安定性 (バイト同一) が土台。

## 軸2: オントロジー — 拡張ではなく整列

koyuのオントロジーへの答えは既に出ている: 巨大なクラス階層ではなく、少数の解釈語+開かれた語彙+導出 (ifc-coverage.mdの「対応の思想」)。だから接続は語彙を増やす方向ではなく、**外の語彙体系への整列**として設計する。

**BOT (W3C Linked Building Data の Building Topology Ontology) への射影。** BOTはkoyuと同じ形をしている — bot:Space、bot:adjacentZone、bot:Interface (二つのZoneの界面!)。koyuのboundaryはbot:Interfaceの、авторable な表面記法だと言える。`koyu rdf` で正準JSONからTurtleを生成する (一方向・生成物): 空間=bot:Space、垂直=bot:Storey階層、境界=bot:Interface、隣接=bot:adjacentZone。SPARQLで問える形になり、リンクトデータ (PLATEAU・不動産ID) と同じ土俵に載る。W3C系の既存語彙に自分を説明させることは、論文でのオントロジー上の位置づけそのものになる。

**外部辞書への参照は属性のレイヤー。** 室用途コード (日本の建築基準法系・BOMA・Uniclass・bSDD) への対応は `dict:` 系属性または対応表ファイル (vocabulary-map.muro的なレイヤー) で持ち、コアの語彙は増やさない。「同じ意味に別の語を使わない」台帳の規則を、外の辞書との間でも守る。

**台帳の機械可読化。** spec/vocabulary.md (人間の契約) から vocabulary.json (機械の契約) を生成し、MCPの `model_summary` やエージェントのプロンプトに同梱する。エージェントが語彙の解釈可否を確かめてから書けるようになる。

## 軸3: デジタルツイン — パスがジョインキーになる

原稿の言葉は「実測を設計の上に重ねる一枚のレイヤー」。合成 (ADR-0010) が実装された今、これは文字通りレイヤーの話になる。ただし二種類を区別する:

**静的な実測 (as-built) は上書きレイヤー。** 竣工時の実測寸法・確定仕様は `as-built.muro` として合成する — これはADR-0010で見送った「上書き合成 (レイヤー強度)」を最初に要求するユースケースになる。設計値と実測値の差分そのものが品質記録である。

**動的な状態 (センサー・稼働) は原本に入れない。** 温度・人感・予約・電力は時系列であり、git-textの原本に書くものではない。koyuの貢献は**アドレス空間**である: `/L5/A/ldk` という人間可読パスが、IoTプラットフォーム・BEMS・予約システムの外部キーになる。GUIDの対応表が要らない — パスがそのまま意味である。実証は小さくできる: 室ごとの計測値CSV (path,timestamp,value) をugatsuの平面にオーバーレイする一枚。

**グラフは行動の地図。** passable() の空間グラフ+扉の位置は、そのままロボット・避難シミュレーション・フィジカルAIのナビゲーションの骨格である (原稿の予告)。メッシュから歩行グラフを復元する業界の逆問題を、koyuは原本が既に持っている。

**都市への接続は測地から。** polygon (ADR-0011) で敷地の形は入った。残るのは測地座標と真北 (MUN-159) — 一点の基準座標+回転角があれば、建物全体がPLATEAU/CityGMLの座標系に載り、zoneが不動産IDに対応づく。日影・斜線・延焼ラインの実計算もこの一歩の先にある。

## 順序と分水嶺

1. **MCP (済)** — エージェントが働ける。分水嶺: エージェントは設計変更をcheck緑のまま完遂できるか (eval)。
2. **rdf/BOT出力** — 小さい (正準JSONの射影)。分水嶺: SPARQLでdoorsと同じ問いに答えられるか。
3. **上書き合成 (as-builtレイヤー)** — ADR-0010の保留の回収。分水嶺: 設計/実測の差分が構成の言葉で出るか。
4. **測地 (MUN-159)** — 都市接続の鍵。分水嶺: PLATEAUの上にtowerが正しく置けるか。
5. **ツインデモ** — パス=ジョインキーの可視化。ugatsuに受け皿。

軸をまたぐ含意: MCPのツール群はそのままオントロジーの「解釈の実装」であり (doorsが答えられることがboundaryの意味論)、ツインの問い (「いまLDKに誰かいるか」) はグラフの問いに時制が付いただけである。三軸は別々の機能ではなく、同じ原本の三つの読み方である — 面積・避難・区画が同じ記述の異なる読み方であるのと同じように。
