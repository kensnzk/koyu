# ADR-0012: MCPサーバー — 建物をエージェントの作業対象にする

日付: 2026-07-23 / 状態: 採用 (v0.8) / 構想: docs/horizon.md 軸1

## 文脈

設計基準「LLMが一つのコンテキストで読み切り、正しく部分編集できるか」は、トークン実測 (一棟8,099トークン、IFC4の1/14) で「読める」ことが数字になった。次は「働ける」— LLMエージェントが建物を読み、問い、編集し、検証する標準の接続点が要る。MCP (Model Context Protocol) はその業界標準であり、Claude・各種エージェントがそのまま話せる。

## 決定

**1. `koyu-mcp` — stdioのMCPサーバーをkoyu本体に同梱する。** ツールは既存の関数の薄い公開に徹する: `model_summary` (要約 — まず呼ぶ)、`check` (門番)、`layers` / `write_layer` (原本の読み書き)、`doors` / `spaces` / `light` / `site` (グラフへの問い)、`plan_svg` (形は生成物)、`canonical_json` (機械形式)。サーバーに意味論は無い — doors が答えられることが boundary の意味論であり、それは koyu の関数が既に持っている。

**2. 依存ゼロを守る — SDKを使わず、JSON-RPC 2.0を手書きする。** MCPのstdioトランスポートは行区切りJSONのJSON-RPCにすぎない (~100行)。パーサを自作した時と同じ賭け: プロトコルの表面積が小さいなら、依存より自前の方が軽い。

**3. ステートレス。** 全ツールが `file` (entry) を受け、毎回 `parseFile` で合成する。towerでも数msであり、セッション状態・キャッシュ・無効化の問題を最初から持たない。原本はファイルシステム、履歴はgit — OwnerHistoryを持たない方針 (原稿) の帰結がそのまま接続の設計になる。

**4. エージェントのループはgitのそれと同型にする。** 読む (layers) → 書く (write_layer) → **checkが門番として通す/弾く** (エラーは出所レイヤー:行つき — 分担の言葉で返る) → 問いで帰結を確かめる (doors/light/site)。write_layer は書き込み直後のcheck結果を返す — 編集と検証を一往復にする。書き込みは `.muro` 拡張子かつentryのディレクトリ配下に限定する。

## 代替案

**別パッケージ (muro-mcp)** はリポジトリを増やす割に、中身はkoyu関数の呼び出しだけ — ugatsuを分けた理由 (変化速度・依存の重さ) がここには無い。**公式SDK採用**は runtime依存ゼロの主張を壊す。**セッション状態を持つ設計** (open→操作列) は、LLMの再試行・並行呼び出しで壊れやすく、ステートレスの数msを買う理由が無い。**編集をパッチ形式にする**案 (行差分) は、レイヤーが十分小さい (最大でも百数十行) ため全置換で足り、失敗モードが単純。

## 帰結と代償

Claude等から `koyu-mcp` を登録するだけで、towerを読んで「Bタイプを割って採光を保つ」型の作業ができる。スモークテストが initialize→問い→壊す→門番が弾く→直す→緑 の全ループを保証する。代償: プロトコル改訂への追随は自前 (表面積が小さいので許容)、resources/promptsは未実装 (空を返す — 必要になったら)。次の分水嶺はeval (horizon.md): エージェントは設計変更をcheck緑のまま完遂できるか。
