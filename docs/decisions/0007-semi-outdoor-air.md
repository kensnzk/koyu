# ADR-0007: 半屋外は導出する — 遮蔽性はair属性、kindにrailを入れない

日付: 2026-07-23 / 状態: 採用 (v0.4)

## 文脈

屋外階段が平面図で屋外に見えない、という指摘から始まった。原因は描画ではなくデータにある: 手すりを `spec:手すり壁` のwallとして書いており、「囲う壁」と「囲わない手すり」の区別をデータが持っていなかった。仕様が書かれていないものは描き分けられない。

最初の実装案は境界kindに `rail` を足すことだった。これは途中で棄却した — **IFCを参照すると、IfcRailingは要素クラスであり、空間境界の関係 (IfcRelSpaceBoundary) が言うのは Physical/Virtual と Internal/External だけである。** kindに物の名を入れ始めると、明日は curtainwall、fence、hedge… と増殖し、退けたはずのクラス階層を関係の側で再発明することになる。

## 決定

**1. kindは関係のトポロジーだけ (wall/open + 垂直)。物の名はspec語彙。** 手すりは `boundary A /out t:120 spec:手すり air:1` — wallである (扉がないから通れないのはwallの既定)。specは自由語彙でツールは解釈しない。

**2. 解釈される性質は `air:1` (遮蔽しない) の一語だけ足す。** 通行可能性はkindが言い、遮蔽性はairが言う。直交する二軸を混ぜない。

**3. 半屋外は宣言ではなく導出する。** 外部 (type:exterior) に対して open または air:1 の境界を持つ空間が半屋外である。バルコニー・屋外階段・ルーフテラスは、手すりを正しく書けば自動的に半屋外になる。

**4. 導出の帰結。** statsでは半屋外を床面積合計から別掲し (算入条件は法規細部のため)、ゾーン面積 (専有面積) からも除く。lightは半屋外越しの窓を0.7掛け (縁側補正に倣う粗い値)。平面図ではair境界を細実線で描く — 壁の黒帯と描き分けることで、囲われていないことが図から読める。「仕様かデザインか」への答え: データが仕様を持ち、描画はそれに従う。

## 代替案

kindにrailを足す案 (上述、実装後に棄却)。空間に `outdoor:1` を宣言する案は、境界を正しく書けば導出できる情報の二重宣言であり、矛盾の置き場になる。

## 帰結と代償

airは遮蔽の有無だけを言う二値で、開放率 (欄干の透過度) や防風・防雨の程度は持たない — 基本計画の解像度。屋根の有無 (半屋外のうち庇下か吹きさらしか) は未分節で、日影・斜線を扱う時に再訪する。
