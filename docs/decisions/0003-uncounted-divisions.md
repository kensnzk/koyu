# ADR-0003: 数える分節と数えない分節

日付: 2026-07-23 / 状態: 採用 (v0.1)

## 文脈

同じ室の中で床材が変わる。同じ境界なのに途中から壁材が変わる。これをどう書くか。最初の案は「仕様が変わるところには分節が潜んでいる — 空間を割れ」だったが、これは要求に合わないと判明した。**室は面積計算の単位である。** 執務室は執務室として一室で扱いたい。採光・排煙・収容人員・貸室面積 — 面積にまつわる計算はすべて室を単位に走る。土間があるからといってホールが二室になってはならない。

## 決定

**分節には二種類ある、と認める。数える分節と、数えない分節である。**

数える分節は室と境界だ。面積表に一行として載り、室数に数えられ、グラフの節点と辺になる。垂れ壁のない分節でも、面積や動線の上で別の単位なら `type:open` の境界で割る — これは今まで通り。

数えない分節は、室に従属する領域 (`area`) と、境界に従属する区間 (`seg`) である。字下げで親に付き、属性の上書きだけを運ぶ:

```
space /L1/hall hall X1..X2 Y1..Y2 name:エントランスホール floor:フローリング
  area X1..X1+1800 Y1..Y2 name:土間 floor:モルタル

boundary /L1/office /L1/corridor t:120 spec:LGS
  door w:900
  seg at:0.75 w:3600 spec:ガラスパーティション
```

**隔離則: 数えない分節は、面積・室数・グラフに一切影響しない。** ホールは土間があっても40.96㎡の一室であり、doorsの答えも隣接も変わらない。これはテストで恒久的に保証する (test/uncounted.test.ts)。segの位置は開口と同じ流儀 (at, w, edge) をとる。

使い分けのリトマス試験は二つ。**「面積表に一行として現れてほしいか?」** — 欲しければ space、要らなければ area。**「そこを通れるかが問題になるか?」** — なるなら boundary/opening、ならないなら seg。

## 代替案

空間を割ってパス階層で束ねる案 (/L1/hall/doma + /L1/hall/main) は、室の同一性と面積単位を壊すため退けた。割る操作はこのデータでは安いが、安いからといって単位が変わってよいことにはならない。属性値を領域つきの複合値にする案 (floor: [{rect, material}]) は、同じ内容の形式化が弱いだけの版である。実施設計まで先送りする案 (基本計画と割り切って捨てる) は、土間や延焼ラインの仕様変化のように基本計画レベルで意味を持つ例がある以上、粗すぎる。

## 帰結と代償

物の記述への滑り戻りを防ぐ防波堤が「幾何を持たせない」から「**構成に影響させない (隔離則)**」に移った。area/segは幾何 (範囲) を持つ — だが構成 (グラフ・面積・室数) からは見えない。この線は今後も試される: areaに面積の内訳を答えさせたくなったら (仕上げ数量など)、それは構成の問いではなく生成物の問いとして実装する。境界の仕様が変わる場所の多くは実際には向こう側の分節 (廊下の相手が受付から執務に変わる等) や外部の分節 (延焼ライン → /outを割る) であり、segはその残余のための最後の道具である — 迷ったらまず「向こう側に分節が潜んでいないか」を疑うのが正しい順序。areaの重なりは現状未定義 (警告もしない)。必要が出たらレイヤー的な later-wins を明文化する。
