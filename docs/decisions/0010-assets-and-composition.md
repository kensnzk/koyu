# ADR-0010: 建具アセットと合成 — Reference/Instance、ファイル分担、明示位置

日付: 2026-07-23 / 状態: 採用 (v0.6) / 対応Issue: MUN-154 (建具語彙) ほか

## 文脈

三つの要求が同じ場所に落ちた。(1) 開口の位置は中点しか置けず、位置を書けるようにすれば「壁からはみ出さないか」の検証が必然になる。(2) 開き戸/引き戸/自動ドアといった建具の型 — Revitの言葉でFamily、USDの言葉でReference/Instance — を、開口ごとの繰り返しなしに書きたい。(3) 空間一次のテキストは、USDのコンポジションのように**ファイル群として分担・合成**できるべきだ — 全体の一貫性はbase層が保ち、各フロアの詳細は各自が書き、一棟へのマージ時にコンフリクトが検査されてビルドされる。

## 決定

**1. 明示位置は通り芯の言葉で書く。** `door D1 at:Y2+1820` — 比率 (`at:0.35`、線分内にクランプ) はそのまま残し、通り芯参照は絶対位置として解釈する。明示位置はクランプしない — 建具幅の半分が線分から溢れれば、許容範囲つきのエラーになる (「中心の許容 450〜3190mm」)。軸も検査する: 水平線分にはX系、垂直線分にはY系。同じ線分上の開口同士の重なりもエラー。位置が言葉になった瞬間に、位置の間違いも言葉で指摘できる。

**2. 建具アセットは参照される既定値の束。** `asset SD1 door w:800 h:2000 style:sliding name:片引き戸` — 開口は先頭トークンで参照し (`door SD1 sill:800`)、インスタンス側の属性がアセットを上書きする (Reference/Instance)。`style` (hinged/sliding/auto) は解釈される語彙で、平面図の建具表現 (開き弧/引き線) が変わる。アセットは第4の要素ではない — 空間・境界・ゾーンの三つ組は増えず、開口の属性の出所が一箇所になるだけである。建具表はこの台帳の一覧にすぎなくなる。

**3. importで合成する。** `import ./L1.muro` — パスは書かれたファイルからの相対。base層 (main.muro) が基盤 — `koyu`版・`name`・`unit`・`grid`・`level` — を一度だけ宣言し、アセット・敷地・各フロアの層を重ねる。合成は加算であり、衝突はビルドエラー: 空間パスの重複・アセット名の重複・grid/nameの再宣言は、両者の出所 (ファイル:行) つきで報告される。同じファイルの二重importは冪等 (一度だけ読む)。**`koyu check main.muro` が一棟のビルドの門番になる。**

**4. 正準JSONは合成後の一棟。** 機械形式にimportは残らない — 合成済みの単一モデルが出る。assetsブロックが加わり、明示位置は書かれた表記 (`"at": "Y2+1820"`) のまま保存される (正準形は語り直さない)。

## 代替案

**USD流のレイヤー強度 (強い層のopinionが弱い層を上書きする) は今は採らない。** koyuの合成は加算+コンフリクト検査であり、同じものを二度書けばエラーである。上書き合成は改修・オプション違いのような差分レイヤーが実際に要求したときに、別のADRとして導入する — 設計の分担にはまず「黙って上書きされない」ことのほうが要る。**ファイルごとの名前空間接頭辞**も要らない — パスの階層がすでに名前空間であり、どのファイルが木のどこを書くかは分担の自由に任せる。**建具を第4要素にする**案は三つ組を壊す割に、得るものが既定値の束と変わらない。

## 帰結と代償

**パーサ本体は純粋になった (v0.6.1)。** 合成の入口は三つ — `parse` (単一ソース)、`parseFiles` (仮想ファイル群 — ブラウザのビューワー向け。キー空間の中でimportを解決)、`parseFile` (fs — node専用の薄い層 `koyu/node` に分離)。checkのエラー・警告も出所レイヤーつきになった (`L1.muro:4行目: …`) — 分担して書いたものは、間違いも分担の言葉で指摘される。

分担が実際に回る: examples/house/ は main (基盤+垂直関係) / assets (建具) / site (敷地) / L1 / L2 の5ファイルで、単一ファイル版 house.muro と同じ答え (92.75㎡) を返す。エラーの文言が設計の言葉になった (「D1 (幅900) が境界線分からはみ出します」)。まだないもの: レイヤー間の上書き (すべて加算)、アセットの入れ子参照、バリアント (USDのvariant — 案の分岐)、フロアファイル単独のcheck (基盤が無いので単独では解釈できない — base込みで検査する)。循環importはseen集合により無限ループではなく冪等になる。
