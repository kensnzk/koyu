# ADR-0002: 高さ方向の一貫性と、通り芯からのオフセット

日付: 2026-07-23 / 状態: 採用 (v0.1)

## 文脈

一フロアを超えて複数レベルを書くと、BIMが「3Dであること」によって暗黙に担保していた課題が表に出る。壁や床が幾何として存在すれば、階高と天井高とスラブの関係は形の整合として自然に成立し、食い込みは目に見える。空間一次のテキストにはその暗黙の担保が無い。どう置き換えるかを決める必要があった。

あわせて位置づけをはっきりさせる。**このデータの解像度は計画初期 — 基本計画レベル — である。** 垂れ壁や建具詳細は表現しない。それは省略ではなく抽象度の選択であり、まさにこの解像度に対して既存BIMは重すぎた (計画初期にBIMが使われない最大の理由)。目指す使い勝手の参照点はVectorworks的な軽さで、ただし原本は図形ではなく構成である。

## 決定

**1. 暗黙の3Dを、宣言された不変量と検査に置き換える。** レベルが `z` (FL高さ)・`h` (基準天井高)・`slab` (その階の床組み厚 = 下階天井面から自階FLまで: スラブ+懐+仕上) を宣言し、check が不変量「下階空間の天井高 + 上階slab ≤ 階高」を検査する。超えれば食い込みエラー。等号でなく不等号にしたのは、天井懐に余りのある設計 (下がり天井) が正当だからで、余り寸法は `levels` コマンド — テキストの矩計 — が見える化する。

**2. 床は書かない。** 垂直の隣接は平面の重なりと隣接レベルから導出され、境界の既定は床 (slab) である。書くのは例外 — 床が繋がる場所 — だけ: `type:stair` (階段、扉0枚で通行可)、`type:shaft` (EV等、空間として連続するが人は通れない)。将来の吹抜けは `type:void` (床が無い=ひとつの空間が階をまたぐ) としてここに加わる予定。水平の境界は全宣言 (書かなければ警告) なのに対し垂直は既定+例外宣言と非対称なのは、水平の分節が設計判断そのものであるのに対し、床の存在は階という概念の既定だからである。

**3. 通り芯からのオフセットは `X2+600` と書く。** 座標を直接書く記法は採らない。芯からの寸法で語ることで通り芯の言語を保ったまま、芯に載らない壁 (コアの間仕切り、水回りのブース壁など) が書ける。

**4. EVシャフトを通行不可としたのは避難の問いに正直であるため。** `doors /L1/ev /L2/ev` は4枚 (階段迂回) と答える。エレベーターは避難経路ではない。

## 代替案

三次元を生成して幾何干渉で検査する案は、検査が生成物の側に移り、エラーが構成の言葉 (「食い込む」) ではなく形の言葉 (メッシュ交差) になるため退けた。原本の検査は原本の言語で行う。空間ごとに床レベル・天井レベルを書く案は冗長で、既定 (レベルの宣言) + 例外 (h属性) で足りる。slabを境界ごとに書く案は、9割が同じ値の反復になりノイズになる。

## 帰結と代償

段差床・スキップフロアはレベルを増やして書くことになる (中2階 = 別レベル)。メゾネットと吹抜けは type:void の設計と同時に再訪する。構面の通層 (上下階で壁が揃っているか) はまだ検査していない — 揃っていない構成も計画としては正当なため、警告ではなく問い (クエリ) として提供するのが筋だろう。斜め壁は直交からの最初の逸脱として、多角形領域と同時に設計する。
