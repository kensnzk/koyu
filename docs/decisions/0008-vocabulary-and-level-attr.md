# ADR-0008: 語彙の台帳と、パス=集計の階層

日付: 2026-07-23 / 状態: 採用 (v0.4)

## 文脈

開き勝手をどう書くかという問いは、掘ると「要素ごとの属性の付け方をどうルール化するか」という問いだった。属性が場当たりに増えると、同じ意味に別の語が使われ、どれをツールが解釈するのか分からなくなる — 開かれた語彙の「開き方」に規則が要る。あわせてメゾネット (階を跨ぐ住戸) がゾーンのパス接頭辞と衝突する問題があった。

## 決定

**1. 語彙の台帳 (spec/vocabulary.md) を契約とする。** 規則は五つ: kindは関係のトポロジーだけ / 物の名はspec語彙 (ツールは解釈しない) / 解釈される属性は台帳に載せ、変えるときはADRを書く / 単位はmm・位置は0..1・キーは小文字英字 / 継承と上書きの規則 (use: zone→space、floor: space→area、spec: boundary→seg) を明文化する。自由な k:v はいつでも書けるが、意味が定まったら台帳に追記して育てる。台帳は一つ。

**2. 開き勝手は解釈される属性の実例として `hinge` と `swing` で書く。** `hinge:W/E` (水平線分) / `hinge:N/S` (垂直線分) が吊元、`swing:a/b` が開く側 (境界のどちらの空間へ)。既定は始端側・a側で、書かなければ今まで通り。checkが線分の向きとhingeの軸の整合を検査し、平面図が従う。

**3. パスの第一義は集計の階層である。** レベルは既定でパス先頭セグメントから読むが、階を跨ぐくくり (メゾネット住戸・吹抜けを含む一体空間の親) は `level:` 属性で階を明示し、パスは所属 (どの住戸か) を言う。`zone /home` の下に `space /home/ldk ... level:L1` と `space /home/bed1 ... level:L2` が並び、住戸=ゾーンが階を跨いで束ねられる (examples/house.ifcxs)。

## 代替案

開き勝手を `door-l` / `door-r` のような開口kindの増殖で書く案は、ADR-0007と同じ理由で退けた (kindはトポロジー、性質は属性)。メゾネットのために「多層空間」(一つのspaceが複数レベルにrectを持つ) を導入する案は、空間とレベルの対応を壊し、高さ検査と平面図の前提を複雑にする — ゾーンが既に「束ねる」道具なので、束ねる側で解くのが筋。

## 帰結と代償

台帳の規律は自己申告であり、CIは未知の属性を警告しない (自由語彙を守るため)。将来、台帳との突き合わせを `check --strict` のような任意検査にすることはあり得る。hinge/swingは平面表現の語であり、建具表 (開き戸/引き戸/折れ戸の別) はまだ持たない — 実施へ向かう情報としてsegやレイヤーの側に置く選択肢を残す。
