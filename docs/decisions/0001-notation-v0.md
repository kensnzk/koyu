# ADR-0001: 記法v0 — 空間一次のDSLと正準JSON

日付: 2026-07-22 / 状態: 採用 (素振り段階の決定。M1で再訪してよいが、覆すときはこの文書を更新する)

## 文脈

原稿 (docs/writing-architecture.md) の最初の一手として、二つの室とその間の扉一つだけを自分の記法で書いた。原稿が予告したとおり、この数十行を書くには「空間の同一性をどう与えるか」「境界をどちらの空間に属させるか」「接続をどう表すか」「パスをどう切るか」を決めざるを得ず、記法はここで生まれた。

## 決定

**1. 空間の同一性は人間可読の階層パスに与える。** `/L1/a` がその空間の名であり同一性である。UUIDは持たない。LLMが一つのコンテキストで読み切り部分編集できること、diffが構成の言葉で読めることを、機械的な同一性の頑健さより優先する。改名は履歴上の出来事としてgitに残る。IFCのGUIDなど外部の同一性が必要になったら `id:` 属性を載せるレイヤーとして後置する。

**2. 境界はどちらの空間にも属させない。** `boundary` は二つの空間パスを結ぶ第一級の関係であり、壁とはこの関係に乗る属性 (厚み・仕様・耐火・遮音) である。壁をどちらかの室の所有物にすると、反対側の室から見た記述が二重になるか非対称になる。関係として持てば、グラフの辺がそのまま得られる。

**3. 接続は境界上の開口として表す。** `door` / `window` は境界の字下げ行に書く。垂れ壁のない開放的な分節は `type:open` の境界とし、扉ゼロで通行できる。これにより「一体空間の中の分節」(リビングとダイニングなど) が、壁のある境界と同じ語彙で書ける。

**4. パスは /レベル/空間 で切る。** パスの先頭セグメントがレベル名ならその階に属する。外部のようにレベルに属さない空間はサイト直下 (`/out`) に置く。粒度 (室の下位のゾーン、室の上位の住戸) はパスの階層で表せる見込みだが、M1まで判断を保留する。

**5. author形式は独自DSL、機械形式は正準JSONの二層構成。** 人とLLMが読み書きする原本は一行一文のDSL。合成 (レイヤー重ね)・diffの土台・外部ツール接続は `ifcxs json` が吐く正準JSONで行う。書き比べと評価は spec/notation-v0.md にある。

**6. 形は生成物。** 壁芯線分は空間の割付から導出され、ソースには書かれない。壁厚は芯振り分け、面積は壁芯算定。「壁を置く」操作はこのデータのどこにも存在しない。

## 代替案

UUID主キー (IFCXの選択) は可読性とLLM編集性を損なうため退けた。壁を物として持つこと (BIMの選択) は記述対象の取り替えという主題そのものの放棄になる。YAMLをauthor形式にする案は、DSLの密度とJSONの機械性の中間にあって決め手を欠いた。

## 帰結と代償

DSLはエディタ支援が無く文法の保守が要る。文法を「一行一文 + key:value + 字下げ開口」から増やさないことで抑える。v0の領域は矩形ひとつでありL字型の室が書けない — M1の一フロアで必ずぶつかるので、そこで多角形または複数矩形を決める。開き勝手は記述せず「先に書いた空間側へ開く」という描画規約に逃げた — 建具語彙とともに再訪する。
