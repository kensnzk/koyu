# 2026-07-23 — 複数フロアと高さの一貫性

## 追記4 — 半屋外・語彙の台帳・メゾネット (深夜)

「屋外階段が屋外に見えない。仕様かデザインか分からない」という指摘は急所だった。原因は描画ではなくデータ — 手すりと壁の区別をデータが持っていなかった。最初、境界kindに`rail`を足しかけて、「それは愚策では。IFCを参考に」と止められた。正しかった: IfcRailingは要素クラスで、IfcRelSpaceBoundaryはPhysical/Virtualしか言わない。kindに物の名を入れるのはクラス階層の再発明。決定は「kindはトポロジーだけ、物の名はspec語彙、遮蔽しない性質はair:1の一語」(ADR-0007)。半屋外は外部へのopen/air境界からの導出になり、バルコニー・屋外階段・テラスが自動で半屋外になる。平面図はair境界を細実線で描く — データが仕様を持てば、図は描き分けられる。

属性の付け方はルール化した (ADR-0008 + spec/vocabulary.md): kindはトポロジー / 物の名はspec / 解釈される属性は台帳が契約 / 単位規約 / 継承と上書き。開き勝手はhinge(吊元)/swing(開く側)。メゾネットは「パスの第一義は集計の階層」に落ちた — level:属性で階を明示し、ゾーンが階を跨ぐ。部分吹抜けは被覆率≥99%のときだけ階跨ぎ天井高を許す (ADR-0006追記)。戸建住宅の例 (house.ifcxs、41行) がメゾネット機構・部分吹抜け・開き勝手の実証。Aタイプにバルコニーが付き、専有面積は不変 (半屋外は数えない)、採光はバルコニー越し0.7掛け。テスト63本。

## 追記3 — 間取り・L字・吹抜け・採光 (夜)

一日の締めに四つを通した。領域は矩形の合併になり (`+` 区切り)、L字のLDKが書ける。共線の外周は一本にマージされ、2矩形をまたぐ掃き出し窓が普通に置ける。`zone` は数える集約 — 幾何を持たず、パス接頭辞で室を束ね、面積は配下の合計、useは継承。Aタイプだけ間取りまで書き、B〜Eは住戸のまま — 粒度の混在こそが基本計画の実態で、割っても「Aタイプ34.8㎡」の言葉は壊れない (ADR-0005)。

吹抜けは「床の不在も境界で書く」に落ちた。上階に type:void の空間、下階と void境界で結ぶと高さ不変量が宣言的に免除される — 境界を消せばエラーが戻ることをテストで保証。床面積不算入・通行不可・平面図は破線対角線。オフィスのエントランスが2層吹抜けになった (会議室は吹抜けに譲った)。例外階は新記法なし: スパンを割ればよい (ADR-0006)。

`light` は最初の法規クエリ。住居系居室の1/7採光 (補正係数なしの甘い判定) で、54㎡のEタイプが1面の窓では足りないことを実際に指摘し、東面の窓が足された。構成を書けば法規の帰結が「何もせず」出てくる、の最初の実例。

斜めは見送りと判断した。雁行・セットバックは合併+オフセットで書ける。真の非直交は矩形の区間代数の全面多角形化で「簡単なら」の範囲外。曲線と同じ側に置き、実例が要求したら一括設計する。

集合住宅183行→114空間・316境界、テスト56本。同期はtarをやめ、変更ファイル直書き+Mac側コミットに切り替え。

二つの仕事をした。ひとつは昨日の問いへの答えとして、同じ二室一扉をIFC4とIFCXで生成した三方比較 (examples/comparison/ — 0.6KB / 7.3KB / 20KB、IFCXの72%はメッシュ座標)。IFC4はThatOpenのweb-ifcでパース検証した。

もうひとつはM1の前半。2フロアの小さなオフィス (廊下+執務室+会議室+コア) を約100行で書いた (examples/office.ifcxs)。記法v0.1として、通り芯オフセット (`X1+3200`)、レベルの高さ宣言 (z/h/slab)、垂直境界 (`type:stair` / `type:shaft`、床は書かず既定) を足した。

核心は高さ方向の一貫性の担保 (ADR-0002)。BIMは3Dであることでこれを暗黙に解決していたが、空間一次のテキストでは「宣言された不変量: 天井高 + 上階slab ≤ 階高」の検査に置き換えた。食い込みはエラー、積み上がりは `levels` コマンド (テキストの矩計) で見える。EVシャフトは「連続するが通れない」ので、`doors /L1/ev /L2/ev` は階段迂回の4枚と正直に答える。テストは27本。

位置づけも定めた: このデータの解像度は基本計画レベル。垂れ壁を表現しないのは省略ではなく抽象度の選択で、計画初期にBIMが重すぎたという既存の弱点の裏側が主戦場。使い勝手の参照点はVectorworks的な軽さ。

次: 集合住宅の基準階 (ゾーン粒度)、吹抜け (`type:void`)、L字領域、その先に斜め (M1.5)。

## 追記2 — 10階建て集合住宅と基準階の反復

内廊下型集合住宅 (1F共用部・2〜9F基準階5戸・10F PH3戸+ルーフテラス・屋上、EV1基+西側屋外階段) を159行で書いた。反復は「基準階は一度だけ書く」機構で吸収 — レベルの等差宣言 (pitch)、パスのスパン展開 (/L2..L9/A)、垂直積層のstack宣言 (ADR-0004)。159行 → 90空間・244境界。checkは警告ゼロで、10フロアぶんの高さ不変量 (L1: 2800+1000=3800 / 基準階: 2400+500=2900 / PH: 2600+600=3200) が一度に検証される。levelsコマンドの矩計が10層きれいに積み上がるのは壮観。

屋外階段は半屋外の初出。上階では手すり壁 (通れない壁) で外気に接し、地上階では type:open で外部に開放 — この使い分けで「9階の住戸から地上まで扉2枚 (玄関+階段防火戸)」「EVシャフトは通れないので階段迂回」に即答できる。/outの分節 (地上と上空) は先送りしたままで、手すり壁はその近似であることをここに正直に記す。専有率69.2% (1,704㎡/2,463㎡)。

例外階 (特定階だけ違う・吹抜けで床が抜ける) はまだ書けない。ADR-0004の帰結に記録し、type:voidと同時に設計する。

## 追記 — 数えない分節

「同じ室の中で床材を変えたい。でも室は割りたくない — 面積計算の単位だから」という問いから、分節を二種類に分けた (ADR-0003)。数える分節 (室・境界: 面積表とグラフに載る) と、数えない分節 (`area` = 室内の領域、`seg` = 境界上の区間: 属性の上書きだけを運ぶ)。隔離則「面積・室数・グラフに影響しない」はテストで恒久保証。ホールの土間、執務室のリフレッシュコーナー、廊下壁のガラス区間をオフィス例に足した。テスト38本。
